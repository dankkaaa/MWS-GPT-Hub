# Eval Cases v1

Этот документ описывает минимальный набор evaluation-сценариев для LLM orchestration MVP.

## Цель

Проверить:

- правильно ли определяется `task_type`
- правильно ли выбирается модель
- правильно ли выбираются инструменты
- правильно ли включается память
- достаточно ли хорош ответ для demo-сценария

## Как использовать

Для каждого кейса нужно проверить:

1. `classifier`
2. `router`
3. итоговый `model_used`
4. `tools_used`
5. `memory_used`
6. качество ответа модели

## Формат проверки

Для каждого кейса фиксируем:

- `id`
- `цель`
- `вход`
- `ожидаемый task_type`
- `ожидаемая модель`
- `ожидаемые инструменты`
- `ожидаемое использование памяти`
- `критерии успеха`

## Кейсы

### EC-01 Обычный текстовый чат

- Цель: проверить базовый сценарий обычного чата
- Вход:
  - `Привет! Коротко объясни, что такое RAG.`
- Ожидаемый `task_type`: `text_chat`
- Ожидаемая модель: `mws-gpt-alpha`
- Ожидаемые инструменты: `[]`
- Ожидаемое использование памяти: `True`
- Критерии успеха:
  - ответ по делу
  - краткое и понятное объяснение
  - без упоминания внутренней оркестрации

### EC-02 Follow-up с памятью

- Цель: проверить, что память может быть использована в обычном диалоге
- Предусловие:
  - в памяти есть факт `Пользователь делает AI workspace на базе OpenWebUI`
- Вход:
  - `Продолжи и скажи, зачем там нужен роутер моделей`
- Ожидаемый `task_type`: `text_chat`
- Ожидаемая модель: `mws-gpt-alpha`
- Ожидаемые инструменты: `[]`
- Ожидаемое использование памяти: `True`
- Критерии успеха:
  - ответ связан с AI workspace
  - модель не теряет предыдущий контекст
  - ответ не выглядит как новый диалог с нуля

### EC-03 Вопрос по файлу

- Цель: проверить routing для файла и RAG-сценария
- Вход:
  - сообщение: `Сделай краткое summary файла`
  - вложение: `file`
- Ожидаемый `task_type`: `file_qa`
- Ожидаемая модель: `qwen2.5-72b-instruct`
- Ожидаемые инструменты: `["retrieve_doc_context"]`
- Ожидаемое использование памяти: `True`
- Критерии успеха:
  - ответ ссылается на контекст файла
  - если контекста недостаточно, модель честно сообщает об этом
  - стиль ответа структурированный

### EC-04 Follow-up по файлу

- Цель: проверить, что после retrieval модель отвечает на уточняющий вопрос
- Предусловие:
  - файл уже был загружен и есть retrieved context
- Вход:
  - `Какие 3 главные мысли в этом документе?`
- Ожидаемый `task_type`: `file_qa`
- Ожидаемая модель: `qwen2.5-72b-instruct`
- Ожидаемые инструменты: `["retrieve_doc_context"]`
- Ожидаемое использование памяти: `True`
- Критерии успеха:
  - ответ связан именно с документом
  - есть краткая выжимка
  - нет слишком общего абстрактного ответа

### EC-05 Summary по ссылке

- Цель: проверить URL-based сценарий
- Вход:
  - `Посмотри https://example.com и дай короткое summary`
- Ожидаемый `task_type`: `url_summary`
- Ожидаемая модель: `qwen2.5-72b-instruct`
- Ожидаемые инструменты: `["parse_url"]`
- Ожидаемое использование памяти: `True`
- Критерии успеха:
  - ответ отражает содержимое страницы
  - короткий и без воды
  - если страница неполная, это отмечено

### EC-06 Web search для свежей информации

- Цель: проверить factual-сценарий с внешним поиском
- Вход:
  - `Проверь в интернете последние новости про OpenAI`
- Ожидаемый `task_type`: `web_search`
- Ожидаемая модель: `qwen2.5-72b-instruct`
- Ожидаемые инструменты: `["web_search"]`
- Ожидаемое использование памяти: `True`
- Критерии успеха:
  - ответ опирается на search context
  - модель не выдумывает факты без инструмента
  - ответ можно кратко показать на демо

### EC-07 Анализ изображения

- Цель: проверить routing для vision-сценария
- Вход:
  - сообщение: `Опиши, что изображено на картинке`
  - вложение: `image`
- Ожидаемый `task_type`: `image_understanding`
- Ожидаемая модель: `qwen2.5-vl`
- Ожидаемые инструменты: `[]`
- Ожидаемое использование памяти: `True`
- Критерии успеха:
  - route уходит в vision model
  - нет ухода в обычный text-only маршрут

### EC-08 Расшифровка аудио

- Цель: проверить routing для ASR
- Вход:
  - сообщение: `Расшифруй это аудио`
  - вложение: `audio`
- Ожидаемый `task_type`: `audio_transcription`
- Ожидаемая модель: `whisper-turbo-local`
- Ожидаемые инструменты: `[]`
- Ожидаемое использование памяти: `False`
- Критерии успеха:
  - route уходит в ASR
  - память не дергается

### EC-09 Генерация изображения

- Цель: проверить routing для image generation
- Вход:
  - `Сгенерируй изображение офисного AI-ассистента в минималистичном стиле`
- Ожидаемый `task_type`: `image_generation`
- Ожидаемая модель: `qwen-image-lightning`
- Ожидаемые инструменты: `[]`
- Ожидаемое использование памяти: `False`
- Критерии успеха:
  - route уходит в image generation model
  - не выбирается обычный текстовый чат

### EC-10 Ручной выбор модели

- Цель: проверить manual override
- Вход:
  - сообщение: `Ответь этой моделью`
  - `mode=manual`
  - `selected_model=llama-3.3-70b-instruct`
- Ожидаемый `task_type`: `text_chat`
- Ожидаемая модель: `llama-3.3-70b-instruct`
- Ожидаемые инструменты: `[]`
- Ожидаемое использование памяти: `True`
- Критерии успеха:
  - router уважает ручной выбор модели
  - остальная логика не ломается

## Сравнение моделей

Для текстовых задач стоит отдельно сравнить:

- `mws-gpt-alpha`
- `qwen2.5-72b-instruct`

На таких кейсах:

- EC-01
- EC-02
- EC-03
- EC-05
- EC-06

### Что сравнивать

- качество ответа
- краткость и ясность
- устойчивость к контексту
- честность при нехватке данных
- скорость ответа

## Что считать достаточным для MVP

MVP можно считать достаточным по orchestration, если:

- все кейсы правильно классифицируются
- все кейсы идут в ожидаемую модель
- tools вызываются только там, где нужны
- память включается и выключается по правилам
- минимум 3 ключевых сценария проходят качественно:
  - обычный чат
  - файл + вопрос
  - ссылка или web search
