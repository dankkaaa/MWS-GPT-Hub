# Контракт Памяти v1

Этот документ описывает интерфейс между слоем оркестрации и backend-слоем памяти/хранилища.

## Цель

Оркестратору нужен простой и стабильный способ:

- получать релевантную память до вызова модели
- сохранять полезную память после ответа модели

## Ответственность

- LLM engineer:
  - решает, когда нужна память
  - решает, как память подмешивается в prompt
  - решает, какие типы памяти полезны для модели
- Backend engineer по памяти/storage:
  - хранит записи памяти
  - индексирует и достает их
  - возвращает memory packet в согласованном формате

## Формат Memory Packet

Backend должен возвращать структуру такого вида:

```json
{
  "chat_summary": "Пользователь строит AI workspace на базе OpenWebUI.",
  "user_facts": [
    "Пользователь является LLM engineer в команде."
  ],
  "preferences": [],
  "relevant_docs": [],
  "ongoing_tasks": [
    "Собрать orchestration MVP."
  ]
}
```

## Получение памяти

Вызывается до финального запроса к модели.

### Request

```json
{
  "user_id": "u1",
  "chat_id": "c1",
  "message": "Продолжи мою прошлую задачу",
  "task_type": "text_chat"
}
```

### Response

```json
{
  "chat_summary": "",
  "user_facts": [],
  "preferences": [],
  "relevant_docs": [],
  "ongoing_tasks": []
}
```

## Запись памяти

Вызывается после финального ответа модели.

### Request

```json
{
  "user_id": "u1",
  "chat_id": "c1",
  "message": "Продолжи мою прошлую задачу",
  "answer": "Вот продолжение...",
  "task_type": "text_chat"
}
```

### Ожидаемое поведение backend

- при необходимости делать краткий summary хода диалога
- извлекать устойчивые факты о пользователе
- извлекать предпочтения, если они реально полезны
- обновлять список текущих задач пользователя
- сохранять ссылки на документы, если в этом ходе использовались файлы или документы

## Политика использования памяти по типам задач

Память должна доставаться для:

- `text_chat`
- `file_qa`
- `url_summary`
- `web_search`
- `image_understanding`

Память не должна доставаться для:

- `audio_transcription`
- `image_generation`

## Требования к storage

Реализация backend может быть любой, но должна поддерживать:

- хранение истории чатов
- хранение summary
- хранение user facts
- хранение document references
- retrieval по `user_id`, `chat_id` и текущему сообщению

## Ожидания для MVP

Для MVP можно начать с простой версии:

- один короткий `chat_summary`
- небольшой список `user_facts`
- небольшой список `ongoing_tasks`
- без сложного ранжирования на v1

## Возможные расширения позже

- vector retrieval для долгосрочной памяти
- confidence score для сохраненных фактов
- TTL для временных фактов
- source provenance для памяти
